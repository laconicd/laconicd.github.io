HEX 코드 대신 OKLCH를 써야 하는 이유

HEX나 RGB 시스템의 한계를 넘어서, 인간의 인지 구조에 최적화된 OKLCH 컬러 모델의 장점과 실무 적용 팁을 정리합니다.

웹 개발에서 색상을 다룰 때 우리는 오랫동안 HEX(#3b82f6)나 RGB(rgb(59, 130, 246)) 시스템에 의존해 왔습니다. 하지만 이 시스템들은 컴퓨터가 색을 생성하는 방식을 그대로 옮겨온 것이라, 사람이 직관적으로 색을 수정하거나 예측하기에는 한계가 명확합니다.

최근 현대적인 브라우저에서 지원하기 시작한 OKLCH는 이러한 문제를 해결해주는 훌륭한 대안입니다.

왜 OKLCH인가?

OKLCH는 세 가지 직관적인 축을 기준으로 색을 정의합니다.

  • L (Lightness): 밝기. 0%에서 100% 사이의 값을 가집니다.
  • C (Chroma): 선명도. 0은 무채색이며, 값이 커질수록 색이 진해집니다.
  • H (Hue): 색상. 0도에서 360도까지의 각도로 색의 종류를 정합니다.

이 구조의 가장 큰 장점은 예측 가능성입니다.

코드 예시: 직관적인 색상 변형

예를 들어, 브랜드의 메인 색상을 기준으로 '밝은 버전'과 '어두운 버전'을 만든다고 가정해 봅시다.

기존 HEX 방식:

:root {
  --primary: #3b82f6;
  --primary-light: #93c5fd; /* 다시 컬러 피커에서 찾아야 함 */
  --primary-dark: #1d4ed8;  /* 관계성을 코드로 알 수 없음 */
}

OKLCH 방식:

:root {
  --primary-h: 250;
  --primary-c: 0.2;
  
  --primary: oklch(65% var(--primary-c) var(--primary-h));
  --primary-light: oklch(85% var(--primary-c) var(--primary-h)); /* 밝기만 20% 업 */
  --primary-dark: oklch(45% var(--primary-c) var(--primary-h));  /* 밝기만 20% 다운 */
}

위 예시처럼 OKLCH를 사용하면 메인 색상의 정체성(Hue, Chroma)은 유지하면서 오직 밝기(L) 값만 조절하여 논리적인 컬러 시스템을 구축할 수 있습니다.

실무에서 체감하는 장점

1. 균일한 시각적 밝기

기존 HSL 방식의 큰 약점은 숫자로 된 밝기가 같아도 색상(Hue)에 따라 우리 눈에 느껴지는 밝기가 제각각이라는 점이었습니다.

  • hsl(60, 100%, 50%) (노란색) -> 매우 밝음
  • hsl(240, 100%, 50%) (파란색) -> 매우 어두움

반면 OKLCH는 **인간의 인지(Perceptual)**를 기준으로 설계되었습니다. oklch(70% 0.2 60)와 oklch(70% 0.2 250)는 색상은 다르지만 우리 눈에는 거의 동일한 밝기로 느껴집니다. 이는 텍스트 가독성을 위한 대비(Contrast ratio)를 계산할 때 엄청난 편의성을 제공합니다.

2. 프로그래밍하기 좋은 구조

우리 블로그의 태그 시스템에도 적용했듯이, 특정 알고리즘을 이용해 색상을 자동 생성할 때 매우 유리합니다.

// 태그 이름에 따라 색상만 다르게 배정하는 예시
const color = `oklch(70% 0.15 ${hash(tagName)})`;

채도와 밝기를 고정해두고 색상 각도(H)만 흩뿌려주면, 전체적인 디자인 톤(일관된 밝기와 선명도)은 유지하면서도 다채로운 색상들을 시스템적으로 생성할 수 있습니다.

3. 더 넓은 색 영역

P3 색 영역을 지원하므로, 기존 sRGB 시스템에서는 표현할 수 없었던 더 깊고 선명한 색상들을 최신 디스플레이에서 온전히 보여줄 수 있습니다.

결론

OKLCH는 단순한 '또 하나의 컬러 함수'가 아니라, 디자이너와 개발자가 소통하고 코드로 색을 관리하는 방식을 한 단계 진화시킨 도구입니다. 복잡한 HEX 코드 조합에 지쳤다면, 이제는 더 논리적이고 인간 중심적인 OKLCH를 기본 컬러 시스템으로 검토해 볼 가치가 충분합니다.
